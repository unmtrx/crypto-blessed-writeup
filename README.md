# Blessed - Hack The Box Crypto

## Overview

`Blessed` is a crypto challenge about two bugs working together:

1. A predictable custom RNG based on P-256 elliptic curve points.
2. A BLS aggregate signature design vulnerable to a rogue-key attack.

The server starts with several verified robots. To reveal the flag, it requires a valid BLS aggregate signature from all verified robots. We do not know the secret keys for the original robots, but the protocol lets us create one known robot, list all public keys, register a malicious public key, and then make the aggregate public key equal to one key we control.

## Main Weaknesses

### 1. Leaky EC-LCG RNG

The server generates robot IDs with:

```python
yield Wn.x >> 32
yield Wn.y >> 32
```

Here `Wn` is a point on P-256, and each new point is computed by adding the generator point `G`.

Only the lower 32 bits of each coordinate are hidden. Since robot IDs expose the upper bits of consecutive curve points, we can recover the missing low bits using lattice reduction. Once we recover one full RNG point, we can continue adding `G` and predict future RNG outputs.

This matters because the zero-knowledge verification uses:

```python
if next(self.rand) & 1:
    ...
else:
    ...
```

So after predicting the RNG, we know every verifier challenge branch in advance.

### 2. Predictable Verification Branches

To verify an untrusted robot, the server asks for 64 rounds of proof. In each round we send a commitment `C`, then the server randomly asks for either:

- `x`, proving `C = x * G1`
- `sk + x`, proving `C = (sk + x) * G1 - pk`

Normally, we cannot prepare one commitment that answers both branches. But because the RNG is predictable, the solver knows the branch before sending `C`, so it sends a commitment tailored to that branch.

### 3. Rogue-Key BLS Attack

BLS aggregate verification checks all verified public keys together:

```python
bls.FastAggregateVerify(agg_pk, b'unveil_secrets', agg_sig)
```

The server lets us join with any public key. After creating one robot, we know one secret key `sk`, and therefore control:

```text
target_pk = sk * G1
```

The solver lists all existing public keys and registers a malicious key:

```text
rogue_pk = target_pk - sum(existing_public_keys)
```

After this rogue robot is verified, the aggregate public key becomes:

```text
sum(existing_public_keys) + rogue_pk = target_pk
```

So a normal signature from our known `sk` verifies as if all robots signed.

## Exploit Flow

1. Connect to the remote service.
2. Use `create` to get one verified robot with a known secret key.
3. Sign `b"list"` with that key and call `list` to get all robot IDs and public keys.
4. Recover the RNG state from the robot IDs using lattice reduction.
5. Build and register a rogue BLS public key.
6. Predict the 64 verification branches and answer them correctly.
7. Sign `b"unveil_secrets"` with the known secret key.
8. Submit that signature to `unveil_secrets`.
9. Receive the flag.

## Lesson

Cryptographic systems usually fail at the boundaries between primitives, not only inside the primitives themselves. BLS signatures are strong, P-256 is strong, and zero-knowledge protocols can be strong, but this challenge combines them in unsafe ways:

- Randomness is generated by a custom, partially leaking construction.
- The proof protocol depends on unpredictable challenges.
- The aggregate signature protocol accepts untrusted public keys without proof of possession.

The result is a complete break.
